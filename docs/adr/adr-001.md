# Architecture Decision Record (ADR)

## Title: ADR-001: Adopt GCP Cloud Run for Application Deployment

- **Id:** ADR-001
- **Date:** 2026-06-03
- **Status:** Accepted

## 1. Context

- The application needs a scalable, reliable, and serverless environment for deployment.
- We want to minimize the operational overhead of managing infrastructure (like VMs or Kubernetes clusters).
- The deployment process needs to be straightforward and integrated with Google Cloud Platform (GCP).

## 2. Decision

- We will deploy the application using Google Cloud Run. This allows us to run containers in a fully managed serverless environment.
- The front-end application will be deployed from source directly using Cloud Build and Cloud Run integration.
- The service will allow unauthenticated access (publicly available).

The exact commands used to configure and deploy the service are as follows:

```sh
# Enable required APIs
gcloud services enable run.googleapis.com cloudbuild.googleapis.com

# Grant Cloud Build service account permissions to deploy to Cloud Run
gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member=serviceAccount:$PROJECT_NUMBER-compute@developer.gserviceaccount.com \
  --role=roles/run.builder

# Deploy the front-end application from source
gcloud run deploy front-end --source ./apps/front-end --region=us-west1 --allow-unauthenticated

# Allow public invocation (unauthenticated access) for the service
gcloud beta run services add-iam-policy-binding --region=us-west1 --member=allUsers --role=roles/run.invoker front-end
```

> We must allow the "Domain restricted sharing" organization policy on project "pet-routine" in GCP so it is possible to access the service from any domain.

## 3. Consequences (Trade-offs)

### Positive

- Serverless architecture reduces operational overhead and scales automatically from zero to handle high traffic.
- Direct deployment from source using `gcloud run deploy --source` simplifies the CI/CD pipeline initially.
- Pay-as-you-go pricing model is cost-effective for variable workloads.

### Negative / Risks

- Vendor lock-in with Google Cloud Platform.
- Cold starts may introduce latency for the first request after a period of inactivity.
- Publicly exposing the service (`--allow-unauthenticated`) means we rely on application-level security and authentication if needed in the future.
