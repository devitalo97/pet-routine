# Architecture Decision Record (ADR)

## Title: ADR-002: Adopt Google Cloud Artifact Registry for Docker Images

- **Id:** ADR-002
- **Date:** 2026-06-06
- **Status:** Proposed

## 1. Context

- The application is deployed using Google Cloud Run (as established in ADR-001).
- Deploying directly from source combines the build and deployment phases, which can make the deployment process slower and harder to track.
- To improve our CI/CD pipeline, we need a robust, centralized location to store and manage our container images.
- We want to decouple the build process from the deployment process to enable better versioning, image reuse, vulnerability scanning, and faster rollbacks.

## 2. Decision

- We will adopt Google Cloud Artifact Registry as our central container image repository.
- Our deployment workflow will change to:
  1. Build the Docker image.
  2. Push the built image to a repository in Artifact Registry.
  3. Deploy the application to Cloud Run using the specific image tag stored in Artifact Registry.

Example configuration and deployment steps:

```sh
# Enable Artifact Registry API
gcloud services enable artifactregistry.googleapis.com

# Create an Artifact Registry repository for Docker images
gcloud artifacts repositories create pet-routine-repo \
  --repository-format=docker \
  --location=us-west1 \
  --description="Docker repository for pet-routine images"

# Authenticate Docker with Artifact Registry
gcloud auth configure-docker us-west1-docker.pkg.dev

# Build and push the image
docker build -t us-west1-docker.pkg.dev/$PROJECT_ID/pet-routine-repo/front-end:latest ./apps/front-end
docker push us-west1-docker.pkg.dev/$PROJECT_ID/pet-routine-repo/front-end:latest

# Deploy to Cloud Run using the pushed image
gcloud run deploy front-end \
  --image=us-west1-docker.pkg.dev/$PROJECT_ID/pet-routine-repo/front-end:latest \
  --region=us-west1 \
  --allow-unauthenticated
```

## 3. Consequences (Trade-offs)

### Positive

- **Decoupled CI/CD:** Separates the build process from the deployment process, allowing images to be built and tested before being deployed.
- **Faster Deployments and Rollbacks:** Cloud Run deployments are faster when pulling a pre-built image, and rolling back is straightforward by deploying an older, tagged image.
- **Security:** Enables the use of Artifact Registry's built-in vulnerability scanning for Docker images.
- **Centralized Storage:** Provides a single source of truth for all built container images.

### Negative / Risks

- **Increased Complexity:** The deployment pipeline now requires explicit build, tag, push, and deploy steps compared to deploying directly from source.
- **Storage Cost:** Introduces additional costs for storing images in Artifact Registry, requiring lifecycle management to clean up old images over time.
- **Vendor Lock-in:** Further reliance on the Google Cloud Platform ecosystem.
